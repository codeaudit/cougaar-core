Overview of this Release

Build 8.3 is an experimental point release along the COUGAAR 8.0 development branch. Notable additions for this release include:
  • New usage patterns for plugin development that directly interact with the 8.x component infrastructure
  • Sample implementation of a PluginBinder wrapping Blackboard services
  • The capability to load binders and components specified in "ini" files
  • Support at the Node level for implementing additional services and component managers
  • Further componentization of the agent infrastructure
  • Further refactoring of MessageTransport
Build 8.2.1 is a patch release associated with the COUGAAR 8.2 release. Notable additions for this release include:
  • An expanded test configuration: configs/small-135
  • Partial refactoring of MessageTransport
  • Bug fixes
Build 8.2 is a point release along COUGAAR 8.0. Notable additions for this release include:
  • DelayedLP actions which allow batching of things like Notifications
  • New component architecture APIs 
  • Component APIs and implementations of PluginBinder, MessageTransportBinder and BlackboardService
  • Dynamic Component API
    • Build 8.0 is a major release of COUGAAR. Essential differences between 7.2 and 8.0 include: 
    • Support for JDK 1.3.
    • Minor naming changes. 
    • Additional patches.

    Organization of a Release 

    The files in an COUGAAR release fall into the following organizational areas:
    • "core:" domain independent agent architecture
    • "glm:" LDM and PlugIns which have a logistics domain focus based on the ALP program objectives
    • "utility:" GSS and PlugIn Contracts
    • "tools:" scalability and test tools
    • "tutorial:" non-logistics sample societies and PlugIns
    • "server:" centralized society initiation and management for testing, integration and demonstration
    The COUGAAR release will contain the following zip files:
    • cougaar.zip:  COUGAAR demo runtime - all built jar files, configs, data, bin, minimum doc.
    • cougaar-support.zip:  All non-built supporting .jars (unpacks into "sys" subdirectory where Bootstrapper will find them).
    • cougaar-api.zip:  the javadoc for all the main cougaar classes in one place. unpacks to api/* instead of under doc as in previous releases.
    • *-dev.zip:  sources, docs (including javadoc) for each "public" module.
    • md5sums.txt:  md5 checksums of all the zips.
    • build.out: output of the build process
    Notes:
    1. You'll need cougaar.zip and cougaar-support.zip (the first time) to get a
    runnable system.  The rest are extras.  Furthermore, the cougaar-support
    contents do not change very often.

    2. the zips are designed to be unpacked in the same directory, e.g. on
    unix:
            mkdir cougaar
            cd cougaar
            unzip /tmp/cougaar.zip
            unzip -o /tmp/cougaar-support.zip
            setenv COUGAAR_INSTALL_PATH `pwd`

    Test Harness Example Implementations

    We only maintain society configurations that we are currently using for testing purposes and for  COUGAAR demonstrations. These society configurations are available in subdirectories under "configs" in the release. To execute these COUGAAR societies, these nodes can be executed on a single computer or distributed across multiple computers. 

    One society, minitestconfig will work "out of the box". We intend to expand this test society in subsequent releases in size and complexity to test new infrastructure features and to provide reference examples of components component and service design patterns. Plugin and component developers are encouraged to use minitestconfig as a starting point for testing their software for compliance with the latest infrastructure release. 

    In addition, there is a much larger configuration in configs/small-135. The default configuration supports running a relatively small but interesting two node society distributed across two machines.  See the small-135.readme.html file in the configuration directory for more information. 

    Delivering Software Products 

    Please see the Integration Information Area that can be found on the UltraLog web site.

    All PlugIn software must use the Java system parameter "org.cougaar.install.path" or the environment variable COUGAAR_INSTALL_PATH to find data files, scripts, and external programs. For example, if your product is "tops," your classes will be in: 

    "$COUGAAR_INSTALL_PATH\lib\tops.jar"
    and your java code could find your data files with: 
    Properties props = System.getProperties();
    String path = props.getProperty("org.cougaar.install.path") +
                  props.getProperty("file.separator") + "tops" +
                  props.getProperty("file.separator") + "myData.dat";

    Known bugs and other issues

    • The NodeServer module (server.jar) does not currently work as an executable jar file because of an unresolved dependency on core.jar. Until this is fixed, an appserver instance will have to have a specified classpath which includes core.jar.
    • The TOPS TPFDD viewer fails to display aggregate TPFDDs. Until this is fixed in the next release, a workaround is to edit runNewTPFDDShell and change the setting of useDerivedTables to false.
    • COUGAAR components now are all required to use a consistent set of 3rd party jar files. This required resolving a number of conflicts, especially with respect to various openmap versions. Cougaar now specifies openmap.jar version 4.2 (included in cougaar-support.zip). All modules except for UIFramework have been updated appropriately. Expect UIFramework to be fixed for the next release.
